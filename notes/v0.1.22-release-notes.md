# v0.1.22

## Features

### AI tracing across Vercel AI SDK, OpenAI, Anthropic, and OpenRouter-compatible clients

Blyp now emits structured `ai_trace` records for AI workloads through multiple integration styles:

- `@blyp/core/ai/vercel` adds middleware and model wrappers for Vercel AI SDK `generateText()` and `streamText()`
- `@blyp/core/ai/openai` wraps OpenAI SDK `responses.create()` and `chat.completions.create()`
- `@blyp/core/ai/anthropic` wraps Anthropic `messages.create()`
- OpenRouter is supported through the OpenAI-compatible wrapper path
- `@blyp/core/ai/fetch` adds optional low-level transport tracing for status codes, latency, request IDs, and inspected JSON responses

In practice, AI calls now produce normalized traces with provider, model, operation name, usage, finish reason, timing, first-chunk latency for streams, and best-effort tool-call events.

Why it matters:

- AI calls can now be queried with the same Blyp workflows used for request logs and structured logs
- AI traces inherit Blyp's active request-scoped logger automatically, so model activity can be correlated with the surrounding HTTP request
- content capture remains opt-in, so prompts, outputs, reasoning, tool payloads, raw provider payloads, and stream events stay off unless explicitly enabled

Migration:

- install the optional SDK used by the AI entrypoint in your app, such as `ai`, `openai`, or `@anthropic-ai/sdk`
- no migration is required if you are not using the new AI tracing APIs

### End-to-end trace IDs across request, client, connector, and database flows

Server frameworks now generate a Blyp trace ID per request, attach it to request-scoped logs, return it in the `x-blyp-trace-id` response header, and propagate it into downstream logs and persisted records.

This covers framework adapters including Express, Fastify, Hono, Elysia, NestJS, Next.js, Astro, Nitro, React Router, SvelteKit, and TanStack Start. Client logs can also carry the trace ID, and database rows now persist it.

Why it matters:

- browser logs, request logs, connector-forwarded logs, and database records can be correlated without an app-defined correlation field
- frontend and support tooling get a stable ID to surface to users or attach to bug reports

Migration:

- no required migration
- if you want browser-originated logs to join the same trace, pass the request trace ID into `createClientLogger({ traceId })`

### Durable connector delivery queue with SQLite retries

Server-side connector forwarding can now use an internal delivery queue with in-memory buffering plus optional SQLite persistence.

Behavior added in this release:

- retryable connector failures can spill into a Blyp-managed SQLite queue
- retries use configurable backoff, batch sizes, concurrency, polling, and overflow strategy
- Blyp writes queue state under `.blyp/` by default and ensures that directory is ignored in `.gitignore`
- this applies to server connector delivery without Redis or another external queue

Why it matters:

- temporary Better Stack, Databuddy, PostHog, Sentry, or OTLP outages no longer have to mean dropped logs
- single-host apps can get more durable delivery semantics with minimal setup

Migration:

- opt in with `connectors.delivery.enabled`
- if you enable it, plan for a local SQLite file, defaulting to `.blyp/connectors.sqlite`

### Queue inspection and dead-letter helpers for operational tooling

The connector queue now exposes management helpers for status inspection and recovery:

- delivery status tracking records last success, last failure, pending counts, and last error per connector target
- exhausted jobs move to dead letters instead of disappearing silently
- helper APIs expose queue path discovery, queue existence checks, dead-letter listing, retry, and clear operations

Why it matters:

- failed deliveries become visible and recoverable
- Studio and custom tooling can inspect connector health and replay dead-lettered jobs

## Bug Fixes

### `flush()` now waits for durable queue replay before finishing

Connector `flush()` previously could finish before the durable SQLite queue had been fully drained. It now blocks until queued durable work is processed, including staged writes and replayed jobs.

Why it matters:

- shutdown hooks, tests, and one-shot processes can rely on `flush()` actually emptying the delivery queue instead of returning early

### Fastify trace headers are now present even for failures before `preHandler`

Fastify now sets the Blyp trace header earlier in the request lifecycle, so requests that fail in `preValidation` still return `x-blyp-trace-id`.

Why it matters:

- early framework failures remain traceable instead of losing correlation data

### NestJS request trace propagation is consistent

Nest now preserves and exposes the trace ID on the request object, response header, and emitted log records more reliably.

Why it matters:

- Nest integrations now match the rest of Blyp's framework adapters for request correlation

### Structured request logs no longer allow conflicting trace IDs to override the active request trace

Request-scoped structured logs now preserve the real active request trace ID even if default structured fields contain a conflicting `traceId`.

Why it matters:

- correlation stays trustworthy in custom structured logging flows

## Security & Privacy

### Automatic redaction now runs across Blyp log outputs

Blyp now sanitizes sensitive data before it reaches console output, file logging, database writes, connector forwarding, client ingestion forwarding, and framework request logs.

Built-in protections include:

- default redacted keys such as passwords, tokens, cookies, API keys, session values, SSNs, and card fields
- pattern-based redaction for bearer tokens, JWTs, common API key formats, and Luhn-valid card numbers
- configurable extra redaction by key, path, and regex pattern

Why it matters:

- sensitive values are removed before they enter Blyp sinks, rather than relying on downstream masking

Migration:

- no migration required
- use `redact.keys`, `redact.paths`, or `redact.patterns` when you need custom masking rules

## Packaging & DX

### Connector and framework-specific runtime dependencies are now truly optional

Connector packages such as `@logtail/node`, `@databuddy/sdk`, `posthog-node`, `@sentry/node`, and OTLP packages were moved out of default runtime dependencies and into optional peers. Connector and some framework entrypoints are loaded lazily with clearer install-time errors.

Why it matters:

- a plain `@blyp/core` install is lighter and avoids pulling connector-specific transitive trees unless you use them
- users get explicit install hints when invoking an integration without its peer dependency

Migration:

- if you use Better Stack, Databuddy, PostHog, Sentry, OTLP, or certain framework-specific entrypoints, make sure the relevant peer packages are installed in your app
- if you relied on those packages being brought in transitively by `@blyp/core`, you now need to add them directly

### Published package surface now points directly to built `dist` outputs

Blyp now ships public JS and declaration entrypoints directly from `dist`, with updated `exports` and `typesVersions`, instead of maintaining separate shim directories.

Why it matters:

- package resolution is simpler and generated type surfaces are closer to what is actually published
- packaging drift between source and published artifacts is reduced

Migration:

- no change for normal subpath imports like `@blyp/core/express` or `@blyp/core/ai/vercel`
- if you were relying on internal `exports/` or `types/` shim paths, switch to documented public package subpaths

### Elysia declaration surface no longer leaks Elysia's concrete plugin type

The Elysia logger now returns an opaque exported plugin type in declarations, preventing type-surface issues caused by duplicate local Elysia instances.

Why it matters:

- local-path and linked installs are less likely to break TypeScript when the app and Blyp resolve different Elysia type instances

### Stability and security policy documentation

This release adds published `STABILITY.md` and `SECURITY.md` documents.

Why it matters:

- public APIs, deprecation expectations, support scope, and vulnerability reporting now have an explicit contract for adopters

### Release pipeline hardening

Publishing now includes production dependency audits and npm provenance attestation.

Why it matters:

- maintainer release workflow is stricter and published artifacts carry stronger supply-chain signals
